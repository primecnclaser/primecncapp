━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

1A. WHERE "40 DAYS" IS IMPLEMENTED
────────────────────────────────────────────────────────────────────────────────
There are TWO separate lock systems in the codebase that both use 40 days but
are completely independent of each other:

  SYSTEM A — Invoice-date-based lock (_isInvoiceLocked)
    Applies to: excel_invoices rows (imported/legacy invoices in the DB tab)
    Logic:      Lock if the invoice DATE is 40+ calendar days old.
    Key lines:  5661–5673

  SYSTEM B — Entry-save-date-based lock (_canEditEntry / _shouldShowPen)
    Applies to: Finance tab daily entries (finance_ledger)
    Logic:      Lock if 40+ days have elapsed since the entry was FIRST SAVED
                (falls back to entry_date if no local meta exists).
    Key lines:  18692–18741


1B. EXACT LOCK CALCULATION CODE
────────────────────────────────────────────────────────────────────────────────

── SYSTEM A (Invoice date lock) ── lines 5659–5673 ──

  // ── INVOICE AGE — generic days-old check for any invoice/row date ──
  // Accepts any format normalizeDate understands (YYYY-MM-DD, DD.MM.YYYY, DD/MM/YYYY)
  function _invDaysOld(dateStr) {
    if (!dateStr) return 0;
    var iso = normalizeDate(String(dateStr).trim());
    var m = iso.match(/^(\d{4})-(\d{2})-(\d{2})$/);
    if (!m) return 0;
    var y = parseInt(m[1]), mo = parseInt(m[2])-1, d = parseInt(m[3]);
    if (isNaN(y) || isNaN(mo) || isNaN(d)) return 0;
    return Math.floor((Date.now() - new Date(y, mo, d).getTime()) / 86400000);
  }
  // Returns true if Total Bill / Received / Discount edits should be locked for this invoice date
  function _isInvoiceLocked(dateStr) {
    return _invDaysOld(dateStr) >= 40;
  }

── SYSTEM B (Finance entry save-date lock) ── lines 18701–18726 ──

  function _shouldShowPen(type, cat, entryDate, localId) {
    // Salaries: never editable
    if (type === 'monthly_expense' && cat === 'Salaries') return false;
    // Entries on or before May 12 2026 are already locked
    if (entryDate && entryDate <= '2026-05-31') return false;
    // 40-day window check
    const { canEdit } = _canEditEntry(localId || '', entryDate || '');
    return canEdit;
  }

  function _canEditEntry(localId, entryDate) {
    const meta = _finEditMeta[localId];
    if (meta) {
      const daysSinceSave = (Date.now() - meta.firstSavedAt) / (1000 * 60 * 60 * 24);
      if (daysSinceSave > 40) return { canEdit: false, reason: '🔒 Edit window expired (40 days have passed)' };
      const daysLeft = Math.ceil(40 - daysSinceSave);
      return { canEdit: true, reason: `✏️ Edit ${meta.editCount+1} — ${daysLeft}d remaining` };
    }
    // No local meta — use entry_date to enforce 40-day rule
    if (entryDate) {
      const entryMs = new Date(entryDate).getTime();
      const daysSinceEntry = (Date.now() - entryMs) / (1000 * 60 * 60 * 24);
      if (daysSinceEntry > 40) return { canEdit: false, reason: '🔒 Edit window expired (40 days have passed)' };
    }
    return { canEdit: true, reason: '' };
  }

  // (Additional hard cutoff: entries dated 2026-05-31 or earlier are always locked
  //  regardless of the 40-day window — enforced inside _shouldShowPen line 18705)

1D. WHAT FIELDS / ACTIONS DOES THE LOCK AFFECT?
────────────────────────────────────────────────────────────────────────────────

  LOCKED after 40 days (per field):

  ┌──────────────────────────────┬────────────────────┬────────────────────────────────────────────────────┐
  │ Field                        │ Locked?            │ Notes                                              │
  ├──────────────────────────────┼────────────────────┼────────────────────────────────────────────────────┤
  │ Total Bill (total_bill)      │ YES — after 40d    │ DB tab inline edit hidden; RP tab ✏️ btn hidden    │
  │                              │ from invoice date  │                                                    │
  ├──────────────────────────────┼────────────────────┼────────────────────────────────────────────────────┤
  │ Received / Payment entries   │ YES — after 40d    │ RP tab: 🔒 Locked shown instead of ✏️ Edit btn.   │
  │                              │ from created_at    │ Clock runs from payment created_at, not inv date.  │
  ├──────────────────────────────┼────────────────────┼────────────────────────────────────────────────────┤
  │ Discount                     │ NO — always open   │ Discount rows in RP tab always show ✏️ Edit.       │
  │                              │                    │ Confirmed by UI message on line 15819.             │
  ├──────────────────────────────┼────────────────────┼────────────────────────────────────────────────────┤
  │ Bad Debt                     │ NO — always open   │ Bad debt rows in RP tab always show ✏️ Edit.       │
  │                              │                    │ Confirmed by UI message on line 15819.             │
  ├──────────────────────────────┼────────────────────┼────────────────────────────────────────────────────┤
  │ Finance ledger entries       │ YES — after 40d    │ System B (separate). Pen icon hidden.              │
  │ (finance_ledger)             │ from firstSavedAt  │ Salaries never editable. Pre-May-13-2026 locked.  │
  └──────────────────────────────┴────────────────────┴────────────────────────────────────────────────────┘

  IMPORTANT NOTE — Two different clocks for the same "40 days":
    • Invoice Total Bill lock  → measured from INVOICE DATE (when invoice was created)
    • Payment entry lock       → measured from PAYMENT created_at (when payment was recorded)
    These are two separate timers. A payment added 39 days after a locked invoice
    is still editable for 40 days after IT was added.

  EXCEPTION — "imported" payment source:
    Line 15752: if (p.source === 'imported') → always shows ✏️ Edit regardless of age.
    Historical imported payments bypass the 40-day lock entirely.

  EXCEPTION — invoice_ref payments:
    Line 15753-15754: if (p.invoice_ref) → shows 🔒, message says
    "Recorded from invoice X — edit the invoice to change this amount".
    These are not locked by time; they are locked by source (must edit via invoice).

2C. IS THERE AN "INVOICE LOCK SETTINGS" AREA ALREADY?
────────────────────────────────────────────────────────────────────────────────
  NO. There is no UI for configuring the lock duration (40 days).

  The existing "Invoice Setting" section (in General Setting sub-tab) is the
  most logical home for a "Lock Duration" control, as it already owns invoice
  behavior settings. It would need a new input field added to the existing
  section, wired to saveInvoiceSettings() / a localStorage key.

  Alternatively, a new "🔒 Lock Policy" sub-section could be added to the
  General Setting panel without creating a new sub-tab.

AUDIT SUMMARY — KEY FINDINGS
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

  1. TWO LOCK SYSTEMS exist, both using "40" hardcoded:
       System A: invoice date → controls Total Bill + payment-entry edit in DB/RP tabs
       System B: firstSavedAt timestamp → controls Finance tab entry edits

  2. "40" is NOT a constant anywhere — it is a raw literal scattered across
     at least 7 logic lines and numerous UI message strings.

  3. DISCOUNT and BAD DEBT are EXEMPT from the 40-day lock. They can always
     be edited from the Receive Payment tab (by design, per UI messages).

  4. "imported" source payments also bypass the lock permanently.

  5. There is a HARD DATE CUTOFF in System B: entries dated 2026-05-31 or
     earlier are permanently locked regardless of the 40-day rule.

  6. The Settings → General Setting → "🧾 Invoice Setting" section already
     exists (line 3658). Adding a configurable lock duration would fit here
     without requiring a new sub-tab.

  7. To make lock duration configurable, the following 7 logic sites must all
     read from the same config value (localStorage key recommended):
       Lines: 5672, 15736, 15755, 15758, 18715, 18716, 18723
     Plus all UI message strings would need to interpolate the value.
