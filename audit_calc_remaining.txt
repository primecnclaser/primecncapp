CANONICAL FORMULA (what "correct" means below)
════════════════════════════════════════════════════════════════════════

  remaining = (total_bill [+ extra_charges]) - discount - received
              [- received1] - bad_debt

  - `invoices` table (new invoices / database[] array): fields are
    total, extraCharges, discount, received, bad_debt. No received1 —
    this table only ever has one "received" field.
  - `excel_invoices` table (imported/historical invoices): fields are
    total_bill, discount, received, received1 (a SECOND received
    amount, used for partial top-up payments), bad_debt.
  - Ledger-aggregate contexts (customer-level, summed across many
    invoices via the `customer_payments` table) use:
      remaining = totalBilled - totalDiscount - totalPaid - totalBadDebt

A site is marked CORRECT if it includes every term that its table
actually supports. A site is marked INCONSISTENT / BUG if it silently
drops a term (almost always bad_debt) that the same table/row does
support elsewhere in the code.

  - Ledger-aggregate from customer_payments (#13, #14, #17).

    of #18, via `pending_balance`).
  Four different philosophies for deriving the "same" number, spread
  across ~18 call sites, is the root cause of every inconsistency above.

════════════════════════════════════════════════════════════════════════
IDEAL SHARED FUNCTION SIGNATURE
════════════════════════════════════════════════════════════════════════

Two distinct calculation SHAPES actually exist in this codebase and
both are legitimate — they should each have exactly one implementation.

1) Per-row/per-invoice calculation (covers sites #1-12, #15-18):

     function calcRemaining({
       total       = 0,   // total_bill (excel_invoices) or total (invoices)
       extraCharges = 0,  // only used pre-submit, on the invoices side
       discount    = 0,
       received    = 0,
       received1   = 0,   // 0 for the `invoices` table, which has none
       badDebt     = 0
     }) {
       return (total + extraCharges) - discount - received - received1 - badDebt;
     }

   Callers that need a floor for display (never show a negative number
   to the user) apply it explicitly at the call site:
     Math.max(0, calcRemaining({...}))
   Callers that need to detect an OVERPAYMENT / advance credit (as
   rpGetCustomerSummary already does via `advanceCredit`) read the raw
   (possibly negative) result instead of flooring it. Keeping the
   floor out of the shared function itself is what lets both use cases
   share one implementation.

2) Per-customer ledger aggregation (covers #13, #14, #17):

     function calcCustomerRemaining({
       totalBilled   = 0,  // sum of total/total_bill across all invoices
       totalDiscount = 0,  // sum of per-invoice discount + manual discount entries
       totalPaid     = 0,  // sum of 'payment' rows in customer_payments
       totalBadDebt  = 0   // sum of 'bad_debt' rows in customer_payments
     }) {
       return totalBilled - totalDiscount - totalPaid - totalBadDebt;
     }

   rpGetCustomerSummary() and buildAllCustomerSummaries() should both
   call this instead of each re-deriving the same one-line subtraction
   independently (#13/#14 above).

Adopting both would let every site in this report call one of two
functions instead of one of four different ad-hoc calculation styles,
and would make the #4 and #15 bugs (a dropped term, a value computed
but never saved) structurally much harder to reintroduce — a
one-line-formula bug in a shared function is caught everywhere at
once instead of needing to be independently found and fixed in up to
18 separate places.
